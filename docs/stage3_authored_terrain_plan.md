# Stage3 Authored Terrain Plan

## 目的

Stage3 の見た目とプレイフィールを、乱数地形の微調整ではなく、意図を持ったステージ設計へ引き上げる。

このメモは、Stage3 ブラッシュアップの途中で確認した反省と計画を、後続作業で見失わないための戻りどころとして置く。

## 反省

- 直近の修正は、最小変更で安全に進める意識が強すぎて、本来先に考えるべき「良い作品にするための構造変更」を後回しにしがちだった。
- Stage3 の問題は、単なるパラメータ調整や見た目の薄い補正ではなく、地形生成方式そのものが作り込みの制約になっている点にある。
- 「最小範囲の変更」は不具合を防ぐための指針であって、作品品質より優先する制約ではない。
- 趣味プロジェクトでは、スピードや安全性だけでなく、作りたい体験に合わせて設計を変える判断をもっと積極的に行う。

## 現状認識

現在の Stage3 地形は、`data/stages/stage3.json` の `TerrainStrip` に `seed` / `gap_min` / `gap_max` / `center_wave` / `irregularity` などを渡し、seed 付き乱数生成した上下境界を `stage3_composer` が素材で描画している。

この方式は、初期実装としては少ないデータで再現性のある地形を作れる利点があった。一方で、以下のような意図あるステージ設計には向きにくい。

- ある地点に砦やゲートを作る
- ゲート前だけ圧迫感を出す
- 中ボス戦闘エリアを広く取る
- 砲台台や破壊可能ブロックを地形形状と噛み合わせる
- 「狭い」「広い」「蛇行」「開ける」といった展開を明示的に制御する

結論として、Stage3 の作り込みでは、地形トポロジーを乱数生成に任せ続けるべきではない。

## 方針

Stage3 は、乱数地形から手書き制御の地形へ移行する。

ただし、完全なタイルマップやピクセル単位指定にはしない。データ作成コストが高すぎるため、まずは `top` / `bottom` の移動可能領域境界を制御点で指定する方式を目指す。

乱数は以下に限定する。

- 地形素材の選択
- 内部充填の見た目ゆらぎ
- 小物配置の軽いバリエーション

乱数に任せないものは以下。

- 通路幅
- 上下境界の大きな流れ
- 戦闘エリアの広さ
- ゲート、砲台台、破壊可能ブロックの意味ある配置
- ボス前、中ボス前などのステージ展開

## 再計画

### 1. Stage3 地形生成方式の再設計

次の本筋。単なる `stage3.json` の TerrainStrip パラメータ調整ではなく、Stage3 用の authored terrain foundation を作る。

- `stage3.json` に手書き地形データを保持する形式を追加する
- `top` / `bottom` の境界を `x, y` 制御点で指定できるようにする
- 制御点間は補間して、現在の composer が扱える surface run / collision run へ変換する
- 既存の `stage3_composer` 素材描画は可能な限り活かす
- 既存 `TerrainStrip` は他ステージや後方互換のために残す

### 2. Stage3 プレイ空間・戦闘配置見直し

手書き地形の上で、移動可能領域と戦闘配置を一緒に見直す。

- 通路幅、上下比率、蛇行、開ける地点を調整する
- 敵、砲台、floor props、破壊可能ブロック、ゲート周辺を地形と合わせて再配置する
- 地形変更前に敵配置だけ詰めることは避ける

### 3. 地形パーツの役割整理

手書き地形の土台ができた後、素材分類を用途ベースへ整理する。

候補:

- `floor_surface`: 床側の移動可能領域に接する表面
- `ceiling_surface`: 天井側の移動可能領域に接する表面
- `body_fill`: 地中・壁内部の充填
- `exposed_column`: 露出した柱・壁
- `floor_prop`: 床上に積む当たり判定ありの小物
- `decor_prop`: 当たり判定なしの装飾
- `turret_mount`: 砲台台
- `breakable_block`: 破壊可能ブロック

### 4. 破壊可能ブロック専用素材

役割整理後に行う。

- 壊せることが分かる見た目
- 上面ありの配置用素材
- ひび入り
- 弱点や報酬入りが分かる発光差分

### 5. 奥行き・視認性の最終調整

背景と地形の競り合いは最後にまとめて調整する。

#67 で一部調整済みだが、地形方式と戦闘配置が固まった後に再評価する。

## 判断原則

- Stage3 の品質に効くなら、多少大きめの設計変更を恐れない。
- 「最小変更」は安全に進めるための手段であり、目的ではない。
- 作りたいステージの意図をデータ構造で表現できるかを重視する。
- 乱数は雰囲気作りに使い、ステージ設計の主導権は手書きデータに戻す。
- 変更はPR単位では分けるが、PRの小ささを理由に根本課題を避けない。
