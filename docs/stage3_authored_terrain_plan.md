# 手書き地形への移行計画

## 目的

ステージ地形を、seed付き乱数の微調整ではなく、狙ったプレイ体験を直接表現できる手書き設計へ移行する。

このメモは、Stage3ブラッシュアップで確認した反省と計画を、後続PRでも見失わないための戻りどころとして置く。

## 反省

- 直近の修正は、最小変更で安全に進める意識が強く、本来先に考えるべき「良い作品にするための構造変更」を後回しにしがちだった。
- 最小変更は不具合を防ぐための指針であって、作品品質より優先する制約ではない。
- Stage3の問題は、単なるパラメータ調整や薄い見た目補正ではなく、地形生成方式そのものが作り込みの制約になっている点にある。
- 趣味プロジェクトでは、スピードや安全性だけでなく、作りたい体験に合わせて設計を変える判断をもっと積極的に行う。

## 現状認識

従来の `TerrainStrip` は、`seed` / `gap_min` / `gap_max` / `center_wave` / `irregularity` などを受け取り、決定論的な乱数で上下境界を作る。

初期実装としては、少ないデータで再現性のある地形を作れる利点があった。一方で、以下のような意図あるステージ設計には向きにくい。

- ある地点に砦やゲートを作る
- ゲート前だけ圧迫感を出す
- 中ボス戦闘エリアを広く取る
- 砲台台や破壊可能ブロックを地形形状と噛み合わせる
- 「狭い」「広い」「蛇行」「開ける」といった展開を明示的に制御する

結論として、ステージ作り込みでは、地形トポロジーを乱数生成に任せ続けるべきではない。

## 方針

JSON上の地形作成APIは、`TerrainStrip` から `AuthoredTerrain` へ移行する。

`TerrainStrip` は長期的な設計目標ではなく、既存ステージ移行用の旧形式として扱う。残す前提で考えると、乱数パラメータ型と手書き制御型という2つの思想が、composer・preview・docs・testsに漏れ続ける。

ただし、完全なタイルマップやピクセル単位指定にはしない。データ作成コストが高すぎるため、まずは `top` / `bottom` の移動可能領域境界を制御点で指定する方式を採用する。

乱数を使ってよいもの:

- 地形素材の選択
- 内部充填の見た目ゆらぎ
- 小物配置の軽いバリエーション

乱数に任せないもの:

- 通路幅
- 上下境界の大きな流れ
- 戦闘エリアの広さ
- ゲート、砲台台、破壊可能ブロックの意味ある配置
- ボス前、中ボス前などのステージ展開

## 残すもの

`TerrainStripSegment` は、当面は内部表現として残す。

`AuthoredTerrain` の制御点を `TerrainStripSegment` へ変換すれば、既存の衝突判定、表面配置、Stage3 composer描画を活かしながら、JSONの作成体験だけを手書き地形へ移行できる。

名前が実態に合わなくなってきたら、後続PRで `TerrainSegment` などの中立的な名前へ変更する。

## 新しい地形形式

`AuthoredTerrain` は、移動可能領域の上下境界を明示する。

```json
{
  "type": "AuthoredTerrain",
  "theme": "fortress",
  "renderer": "stage3_composer",
  "segment_w": 48,
  "top": [[0, 48], [1200, 78], [2400, 36]],
  "bottom": [[0, 500], [1200, 468], [2400, 520]]
}
```

`top` は上側地形の下端Y、`bottom` は下側地形の上端Yを表す。プレイヤーが動ける領域は `top_y` と `bottom_y` の間になる。

## PR計画

### 1. 手書き地形の基盤

- `AuthoredTerrain` をJSON上の新しい連続地形形式として追加する
- `top` / `bottom` 制御点から実行時地形セグメントへ変換する
- 制御点間は補間し、通路の最低幅を `min_gap` で守る
- Stage3の道中地形を `AuthoredTerrain` へ移す
- spawner、Stage3 composer preview/report、docs、consistency testを更新する
- `TerrainStripSegment` は内部表現として残す

### 2. Stage3のプレイ空間と戦闘配置

- `AuthoredTerrain` の制御点で通路幅、上下比率、蛇行、開ける地点を調整する
- 敵、砲台、floor props、破壊可能ブロック、ゲート周辺を地形と合わせて再配置する
- 地形変更前に敵配置だけ詰めることは避ける

### 3. 地形パーツの役割整理

素材分類を形状だけでなく用途ベースへ整理する。

`tools/stage3_terrain_rects.json` は、画像上の切り出し単位である `groups` と、ゲーム内用途を表す `roles` の二層で管理する。composerや固定地形描画は `floor_surface` / `body_fill` などの role を参照し、`strip_top` / `block_square` などの形状グループ名へ直接依存しない。

候補:

- `floor_surface`: 床側の移動可能領域に接する表面
- `ceiling_surface`: 天井側の移動可能領域に接する表面
- `body_fill`: 地中・壁内部の充填
- `exposed_column`: 露出した柱・壁
- `floor_prop`: 床上に積む当たり判定ありの小物
- `decor_prop`: 当たり判定なしの装飾
- `turret_mount`: 砲台台
- `breakable_block`: 破壊可能ブロック

### 4. 破壊可能ブロック専用素材

- 壊せることが分かる見た目にする
- 上面ありの配置用素材を用意する
- ひび入り、弱点、報酬入りが分かる発光差分を検討する
- Stage3の `fortress_block` は、破壊可能な場合だけ `breakable_block` roleのrect素材を一枚で使い、ダメージ段階に応じて亀裂量を増やす。警告帯・弱点ノード・赤点は足さない
- 縦長ゲートでは小さいsquare素材を無理に拡大せず、`block_tall` を候補に含める。破壊可能ブロックは素材を見切らずに表示し、ブロック寸法を素材アスペクトへ寄せる。高倍率拡大は避け、破壊可能ブロックのHPは既存の約2倍を基準にする

### 5. 奥行きと視認性の最終調整

- 地形方式と戦闘配置が固まった後で、背景の暗さ、地形コントラスト、パララックス、視認性を再評価する

## 判断原則

- `TerrainStrip` を設計目標として温存しない
- 便利な既存実装は、移行を助ける内部部品としてだけ残す
- 作りたいステージの意図をデータ構造で表現できるかを重視する
- 乱数は雰囲気作りに使い、ステージ設計の主導権は手書きデータに戻す
- 変更はPR単位では分けるが、PRの小ささを理由に根本課題を避けない
